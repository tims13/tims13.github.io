apue:Linux下的线程同步

发表于 2020-07-22 分类于 linux

ABOUT

在多个线程共享相同的内存时，如果读取和修改一个共享的变量，因为读取和修改在cpu中无法确保能够在一个CPU周期完成（非原子性的），所以可能会导致多个线程得不到一致的结果，因此就需要线程同步，以避免上述现象。

Linux下的线程同步接口，主要有：互斥量、读写锁、条件变量、自旋锁、屏障。下面将分别结合apue的内容和源码进行总结，作自己的备忘总结。

互斥量

互斥量，本质就是一个锁，确保同一时间只能有一个线程进入临界区，在访问共享资源前要获取锁，在访问完成后释放锁。

互斥量还是比较好理解的，需要注意的是要避免产生死锁，特别是使用多个互斥量的情况下。

读写锁

读写锁类似于互斥量，当其有更高的并行性，其锁分为读锁和写锁，同一时间多个线程可以获取读锁但是只有一个线程能够获取写锁。

为了避免写线程饥饿，一般来说，OS会有下列操作：当有线程已经获取读锁的时候，另外一个线程想获取写锁，OS会让随后的读锁请求阻塞。

一个比较典型的应用是：一个链表被多个线程共享，其查找需要获取rwlock，其修改需要获取wrlock。这样多个线程就能够同时访问链表进行查找，但同一时间只能有一个线程能够对链表进行修改，能够提高访问的并行性。apue的图11-14使用读写锁的例子，就是这个应用，理解比较容易，这里就不贴了。

条件变量

条件变量也是常用的线程同步机制，允许多个线程到达同一个条件。条件变量一般和互斥量一起使用，这样就能够让线程等待特定条件发生。

这里互斥量，能够防止都个线程同一时间请求这个条件变量，防止产生竞争。

执行的过程中，如果一个线程请求条件变量，发现条件为假，其会进入阻塞，然后原子性释放对应的互斥锁。如果条件被改变（另外一个线程signal或者broadcast），在等待条件的线性将被唤醒，然后获取互斥锁，最后再次评估条件，而并非唤醒后就立刻进行评估。

自旋锁

自旋锁和互斥量类似，但是其是让线程一直等待（自旋）来对线程阻塞，因此一般用于锁持有时间短的情况下，因为可以节省调度的时间。

屏障

屏障（barrier）能够让多个线程在屏障处等待，知道所有线程都到达同一点，然后从该处继续执行。

apue的图11-16使用屏障的例子：使用多个线程对一个大数组进行排序，每个线程排序数组的一部分，当所有线程都排序完毕（到达barrier）时，调用一个主线程对其他线性排序的结果进行汇总（归并排序）。

IO复用：select、poll与epoll

发表于 2020-07-20 分类于 计算机网络

ABOUT

IO复用是指，一个程序能够监听多个文件描述符fd，一旦某个fd就绪，就能够通知到程序，然后进行相应的读写操作。

但是，IO复用虽然能够监听多个fd，凡是IO复用的本身是阻塞的。而且当有多个fd同时就绪时，没有额外措施的话，一个程序只能按顺序一次处理其中的一个fd。借助多线程/进程，能够实现并发，同时处理多个就绪的fd，处理并发。

本文主要关心的是linux下关于IO复用的系统调用：select、poll、epoll。

select

  1. nfds：被监听的fd数最大值0，指定监听的fd数。因为fd一般从0开始数，因此其值一般是最大的fd值+1。

  2. readfds、writefds、exceptfds：指向可读、可写、异常等事件对应的fd集合

  3. timeout：设置超时时间。调用失败时，其返回的timeout值是不确定的。

  4. 成功时，返回就绪的fd总数，没有就绪返回0，失败返回-1并设置errno。

select的过程可以总结为以下步骤：

  1. 将fd_set从用户空间拷贝到内核空间。
  2. 内核遍历需要监控的fds，调用相应的poll来检查是否有相应的事件，timeout时间内，如果没有相应的事件，select将保持阻塞。如果在timeout时间内某个fd上面有相应的事件，则将fd_set（rset）对应的位置位，函数返回。

上述过程是用函数调用者的角度总结的select过程，其实在内核中，select背后还有许多逻辑，待以后阅读源码的时候总结。由上述过程可以发现，select主要是把轮询fds，监测其事件的操作交给了内核，他们之间的交流通过fd_set，这决定了select具有以下缺点：

  1. 监听的fds有限。由上可知，FD_SETSIZE限定了fd_set的大小，因此select监听的fds是有限的。
  2. fd_set不可重用，如上代码所示，每次使用前，需要置零。
  3. fd_set在用户和内核之间传递，需要一定的开销。
  4. 采用轮询的方式，内核采用轮询的方式感知事件，用户轮询fd_set来判断哪个fd是就绪的，都采用线性扫描，时间复杂度是O(n)，效率不高。

poll

  1. fds：pollfd结构体：
  1. nfds：指定被监听事件fds的大小。
  2. timeout：超时时间，单位ms。timeout==-1，poll永远阻塞直到监测到事件；timeout==0，poll立即返回。
  3. 返回值同select。

poll与select大体相似，不同之处就是其利用的数据结构pollfd。

poll的调用过程和select大概类似，不过用户和内核的桥梁是pollfd，如果内核在TIMEOUT时间内产生事件，会将对应的pollfd中的revents置位，然后函数返回。

poll的特点如下：

  1. 内核中，底层存放pollfds使用的是链表，因此其没有select那样的对fds的数量的限制。
  2. pollfds相比select，其也需要重新置位，由用户手动设置revents为0，表示初始状态。
  3. 其也有上面select的第3、4个缺点。
  4. 为水平触发，如果产生事件，其会一直报告事件，直至该事件被处理。

epoll的特点如下：

  1. epoll具有水平触发（LT）和边沿触发（ET）模式。上面已经提到，poll属于LT，如果报告的事件不处理，其下一次还会上报事件。而ET模式下，因为其检测到一个fd上面有事件并通知到应用程序后，应用程序必须立刻处理，其后面就不会再通知该事件。这样，系统中就不会充斥着应用程序不关心的，重复的就绪fd，能提高效率。
  2. 底层使用红黑树要存储所有关心的fds，用双向链表来存储就绪的事件。当有事件、fd就绪时，会通过回调函数，把对应的fd添加到就绪链表中。因此可以从上述样例代码看到，接受到事件后，不必遍历所有的fds，只有遍历就绪的fds就可以了，这大大提高了效率。
  3. 没有对fds的数目限制（远远大于1024），具体数目可以通过cat /proc/sys/fs/file-max查看。
  4. 不少博客提到的，用户和内核使用共享内存存储就绪链表，该说法是错误的。其也是通过拷贝来进行的，内核的具体行为在后面的源码阅读中会更加清晰。

总结与对比

  1. select、poll只支持LT，而epoll支持LT、ET，其还支持EPOLLONESHOT，能够减少事件触发次数，提高效率。
  2. select、poll需要遍历所有fds来发现就绪的fd，时间复杂度为O(n)，随fds的数目的增加而线性增长，而epoll只需要遍历就绪的fd来对就绪fd进行处理，能够减少时间复杂度。
  3. select支持的fds数目小，一般限制在1024，而poll、epoll则没有这个限制。
  4. 与内核的交流：select使用一个位图fd_set、poll使用pollfd，epoll使用epoll_event。
  5. epoll虽然看起来最优，但是当连接数少且都十分活跃的情况下，select和poll可能会比epoll的性能好，因为epoll的机制涉及到许多函数回调，会产生一定的开销。
  6. 上述只是select、poll、epoll的从用户的角度的简析，具体内核的相应行为还需关注内核源码，后续进行分析。

HTTPS: HTTP+加密+认证+完整性保护

发表于 2020-07-11 更新于 2020-07-12 分类于 计算机网络

ABOUT

HTTPS面试的时候会考到，现对其进行总结。参考资料：

【硬核】HTTPS原理全解析 必须说一下这个UP讲得真的不错~

《图解HTTP》 - 上野宣

总结

HTTPS

HTTP使用的是明文传输，因此安全性较差，在需要安全性较高的环境下不适用，因此需要安全性更高，这就是HTTPS(HTTP Secure)。

HTTPS不是新的协议，是在HTTP的通信接口处，使用SSL（Secure Socket Layer）和TLS（Transport Layer Security）协议代替。

如上图（摘自图解HTTP），SSL是HTTP和TCP之间的层，其也可以运用在SMTP、Telnet上面。

共享密钥加密（对称密钥加密）

顾名思义，该种加密方式共享一个密钥key：

服务端在制定key时，因为客户端的情况未知，因此其一般只制定一个key，这样的话黑客就也能拿到key，然后截取X，如果再破解f1和f2，则就能轻易地拿到通信数据。

公开密钥加密（非对称加密）

该种加密使用2个key，一个为公钥（public_key），另一个为私钥（private_key）。

公钥是公开发布的，所有人都可以获得。而私钥是保密的，只有服务端自己才有。

当服务端向客户端发送数据Y时，如果黑客窃取Y，因为public_key是大家都能获得的，黑客也可以获得，所以黑客也能够把Y解密成数据data2。

对称密钥加密+非对称密钥加密

由上面可以知道，对称加密因为只有一个key，容易被黑客破解，而非对称加密因为public_key可以被任何人获得，在服务端返回信息至客户端时，容易被黑客破解。

把两种解密结合起来，能够获得一种相对来说更加安全的加密。

客户端先产生一个随机数num1，然后通过非对称密钥加密的方式，将其传送到Server，在Server发送一个确认消息（此处存在简略）后，使用num1作为对称密钥加密的key的方式进行接下来的通讯。因为对称密钥的key是每个客户端生成的秘密随机数，对于每个客户端来说，key是不一样的，因此较上面说的方式安全。

这里第二个步骤中，其实可以仍旧使用非对称密钥加密的方式进行通讯，此时相当于Client和Server均拥有一个public和private key。但是，由于非对称加密的计算开销远远大于对称加密，如果一直使用该种方式，可能会降低服务器的性能，因此综合考虑，第二阶段才采用对称加密的方式。

然而，黑客仍旧能够在这种方式中介入通讯：如果黑客一开始就介入整个通讯，充当服务端和客户端的中间人(MITM, man in the middle），对客户端扮演服务端，对服务端扮演客户端，仍旧可能窃取信息。

CA,Certificate Authority

上面中间人问题的原因，是客户端无法得知，一开始请求的public_key究竟是不是货真价实的，无法确定其是真正的来源自服务器还是来源是黑客，针对该问题，引入了CA。

CA是一个权威的机构，只有经过CA验证的public_key，才是货真价实的public_key。而验证public_key的真实性，需要CA处再经过一层非对称加密。

CA_public_key是写死在OS或者浏览器的，如果不是这样，黑客仍有可能介入和客户端和CA的沟通过程。

经过CA的验证，黑客无法得到CA_public_key也就无法得到public_key，这样即使在后面对称加密方式通信的过程中获取到Y1、Y2，也无法将其解密成真正的信息。

过程总结

上述图示中，为了方便理解，直接省略了一开始商定public_key的过程，其实该过程可以分为下列步骤：

  1. Client->Server：支持的SSL版本、非对称加密算法、随机数R1
  2. Server->Client：确认支持的SSL版本、对称算法、随机数R2、license
  3. Client端进行证书的验证，也就是上图中的验证过程。如果没问题进行下列步骤。
  4. Client->Server：生成随机数R3发送到Server，然后对R1、R2进行散列算法hash，生成数XX发送到Server，即hash(R1,R2) = XX ->Server。
  5. Server端利用之前得到的R1,R2和已经商定的散列算法hash，计算散列后的结果hash(R1,R2)=XX’，如果XX’==XX，说明客户端是可信的，则利用R1、R2、R3来生成public_key，生成的算法也是已经确认过的。
  6. Server->Client：同理，hash(R1,R2,R3) = ZZ -> Client。
  7. Client端收到ZZ，然后hash(R1,R2,R3)=ZZ’，如果ZZ==ZZ’，则说明服务端也是可信的，Client也利用R1、R2、R3使用同样的算法来生成public_key。
  8. 最后，双方都得到public_key，接下来使用该public_key来进行对称密钥加密通信。

二叉树中序遍历的Morris算法

发表于 2020-07-08 分类于 算法

这篇blog清晰且直接地说明了Morris算法的整个流程，本文主要为总结与一些思考。

ABOUT

Morris算法，能够在O(1)空间复杂度实现二叉树的中序遍历。

其主要思想是：找当前节点的前序节点，然后利用前序节点的右空指针，达到中序遍历的效果。因为利用到前序节点的右空指针，所以会对树进行暂时的修改，但是最后会还原。

该算法一个重要前提是：前序节点的右指针一定是空的。（没有设置前）

算法步骤

循环处理各个节点：

  1. 当前节点的左孩子为空，打印（或进行其他处理）然后走向右孩子，该右孩子可能是树原来的右孩子，也可能是前序节点经过设置的右孩子。

  2. 当前节点的左孩子不为空：

    1. 找到当前节点的前序节点pre。

    2. pre右子树为空：说明算法的前面的步骤没有设置pre的右子树，设置pre的右孩子为当前节点，然后走向当前节点的左子树。

      pre右子树等于当前节点：说明当前节点是由pre走向右子树来的，且左子树已经访问完毕，pre的右子树也已经利用完毕（没有利用价值了），需要还原。因此需要：pre的右子树置为NULL，然后打印（或者进行其他处理），走向当前节点的右子树。
